# Mega adversarial review — 2026-06-23 (round 5, whole-codebase)

10 finder lenses → dedup → 3 perspective-diverse skeptics/finding → synthesis. Raised 40, confirmed 29, refuted 11. 131 agents.

> MEGA review verdict: the substrate is solid and well-tested (79 tests pass), but this is NOT "only minor issues remain." Three genuine High-severity defects warrant fixing now: (MR-1) a security/availability regression where one hostile .py aborts the entire index (the I1 RecursionError guard the CodeAdapter has was never ported to the new PythonResolver, which is always-on in the default config); (MR-2) a crash mid-index permanently and silently loses cross-file edges because one logical file ingestion is split across 3 commits with the sha256 skip-token persisted before its edges; and (MR-3) editing a callee-file body silently downgrades a precise 0.97 cross-file CALLS edge to 0.6 and it never recovers, defeating the documented monotonic-confidence guarantee. Two High-severity performance items (MR-4 per-query L2-norm recomputation ~720ms/query at the documented 1e5 ceiling; MR-5 full-corpus materialization with bodies in dirty_nodes, 338MB) are real but were partly tracked/deferred. The remaining ~20 findings are Medium correctness bugs in the new PythonResolver (package __init__ targeting, nested-scope shadowing, sibling-scope dict overwrite, import a.b.c aliasing, submodule attribute calls) plus a cluster of Low spec/doc-drift items. Net theme: the brand-new PythonResolver (feat/python-precise-resolver) carries the bulk of the real defects and its "safe by construction" invariant is violated in several concrete-but-narrow ways; the older substrate is clean. All findings independently reproduced by skeptics and spot-verified against the code.

## Confirmed findings

### MR-1 — High/security — PythonResolver not RecursionError-safe: one hostile .py aborts the whole index (I1 security regression, default config)
**Votes:** 3/3 (all real, High; reproduced end-to-end through default config)  
**Location:** src/memorydb/adapters/code/python_resolver.py:59 (narrow except tuple), 118-141/149-171/198-219 (uncapped recursion); src/memorydb/indexer.py:177 (no per-extractor guard)  

The CodeAdapter was hardened for security I1 with both a depth cap (_MAX_WALK_DEPTH=200, adapters/code/__init__.py:141) and a broad `except Exception` around parse+walk (__init__.py:111-115). The new PythonResolver carries NEITHER: extract() catches only (SyntaxError, ValueError, OSError) at python_resolver.py:59, and _collect_defs/_walk_edges/_scope_locals.walk all recurse with no depth cap. A .py with a deep attribute chain (`a` + `.b`*2000) parses fine then raises RecursionError (a RuntimeError subclass, not in the caught tuple) inside the resolver's own walk; a deeper one makes ast.parse itself raise it. The error propagates through Indexer._extract_all (indexer.py:177, no per-file guard) and aborts the ENTIRE index run, leaving every other file un-indexed. PythonResolver is ALWAYS in ExtractorRegistry.default() (api.py:46, 'stdlib, no extra required'), so a stock MemoryDB.open()/`memorydb index <untrusted-repo>` hits this with no [code] extra. Reproduced end-to-end: 10 good + 1 hostile .py persists 0 of 11 file nodes; the same file leaves CodeAdapter unscathed (returns empty Extraction), proving the asymmetry. Denial-of-index DoS on an operator indexing untrusted code, not RCE.

**Fix:** Mirror the existing I1 remediation: (1) broaden extract()'s except to `except Exception:` (or at minimum add RecursionError) so a bad file yields Extraction() rather than propagating; (2) add a _MAX_WALK_DEPTH cap threaded through _collect_defs, _walk_edges, and _scope_locals.walk; (3) belt-and-suspenders, wrap `ex.extract(path)` in Indexer._extract_all (indexer.py:177) in try/except so no single extractor can abort the run. Add a regression test with a deep attr chain asserting index() completes.

### MR-2 — High/data-integrity — Non-atomic 3-commit index() loses cross-file edges permanently on a crash (sha256 skip-token persisted before its edges)
**Votes:** 3/3 real (2 High, 1 Medium); both crash variants reproduced under SIGKILL  
**Location:** src/memorydb/indexer.py:115-123 (PASS1 commit, file-node sha256), 137 (PASS2 commit), 145 (_resolve_pending commit), 251-252 (gated early-return)  

Indexer.index() commits three times in one run with no outer transaction or try/except spanning the passes. The file node carrying attrs.sha256 — the resumability/skip token read by _existing_files() (lines 196-201) and compared at lines 98-99 — is committed in PASS1, BEFORE that file's edges (PASS2, line 137) and resolved pending edges (line 145). If the process dies between commits (OS kill, power loss, SQLite OperationalError on lock/disk-full, an uncaught error after the first commit), the hash is durably persisted but the edges are not. On the next run the file hash-matches and is SKIPPED, so its edges are never rebuilt. Variant A (crash after PASS1): in-file edges AND pending_edges rows are both lost — total, silent, unrecoverable loss. Variant B (crash after PASS2): the pending row survives but _resolve_pending early-returns (0,0) at lines 251-252 because changed_rels and affected_names are both empty on a clean re-index, so the orphaned-but-resolvable row is never swept. Both reproduced under real SIGKILL. No force/full reindex escape hatch exists (only refresh_embeddings --full); recovery requires deleting the DB. The indexer spec advertises 'resumable' and its crash-recovery note is subtly wrong (it reasons about the symbols commit, overlooking that the same hash gates edges in later passes). One dissent rated Medium (crash-only trigger; spec lists 'resumable-after-crash hardening' as an explicitly unchecked deferred task), but the silent permanent loss plus the crash-independent Variant-B sweep gap sustain High by 2/3.

**Fix:** Make one index() run atomic: wrap all three passes in a single transaction (commit exactly once at the end, rollback on any exception), OR defer the file-node sha256 write until after that file's edges+pending are persisted so the skip-token can never be durable ahead of the data it gates. Independently fix Variant B: in _resolve_pending also sweep orphaned pending rows whose endpoints both now exist regardless of changed_rels/affected_names. Add a force/full reindex flag for Variant-A recovery.

### MR-3 — High/data-integrity — Editing a callee-file body silently downgrades a precise 0.97 cross-file CALLS edge to 0.6, permanently — defeats monotonic-confidence
**Votes:** 3/3 real, High; reproduced end-to-end (before 0.97 / after 0.6)  
**Location:** src/memorydb/indexer.py:203-222 (_delete_file cascade) + 240-267 (_resolve_pending); src/memorydb/store.py:98-106 (monotonic upsert_edge); python_resolver.py:184 (direct 0.97 edge); adapters/code/__init__.py:187 (coarse 0.6 pending)  

store.upsert_edge guarantees monotonic confidence (MAX(existing,new)) so a coarse pass can never downgrade a precise edge — but the incremental indexer defeats it. PythonResolver emits a DIRECT precise edge a.py::caller --CALLS--> b.py::target @0.97 (python-ast); CodeAdapter ALSO leaves a durable pending_edges row (caller, target, CALLS, 0.6) because it can't resolve cross-file. When b.py's body is edited (signature unchanged), _delete_file('b.py') deletes node b.py::target and FK-cascades away the 0.97 edge. a.py is sha-unchanged so its precise edge is NOT re-emitted; `target` is in affected_names so _resolve_pending fires, finds the 0.6 row, and re-upserts at 0.6. The MAX guard is bypassed because the high-confidence ROW was destroyed before re-resolution — nothing to MAX against. The edge stays stuck at 0.6 across repeated callee edits, recovering only if the CALLER file is itself re-extracted (verified: 0.97->0.6->0.6->0.97-after-caller-touch). Impact is real and metric-visible: eval HIGH_CONF=0.9 floor (eval/__init__.py:21,143) drops the 0.6 edge from LOCATE precision_high, and EXPLAIN/ranker weighting silently degrades on the everyday edit-a-callee workflow. The R3L-1 regression test (test_review_r3.py:62) only emits the edge as pending@0.6 and asserts existence (COUNT==1), never confidence — masking this. One dissent noted the edge still EXISTS and points to the correct target (Medium-defensible), but the silent permanent fidelity loss of a contract specifically added during prior adversarial review sustains High by 3/3.

**Fix:** Preserve confidence across re-resolution. Preferred: when _delete_file cascade-removes an edge whose far endpoint survives, persist the surviving precise edge's (src_uid, dst_name, relation, ORIGINAL confidence) into pending_edges before the delete so _resolve_pending rebuilds it at >=0.97; OR record cross-file precise edges in pending_edges too (MAX-confidence) so the durable row carries 0.97; OR re-extract any unchanged file defining a name in affected_names. Extend the R3L-1 fixture to emit a direct high-confidence edge alongside the 0.6 pending row and assert confidence stays >=0.97 after a callee-body edit.

### MR-4 — High/performance — BruteForceVectorIndex recomputes each stored vector's L2 norm on every query (cosine dominates ~700ms/query at 1e5)
**Votes:** 3/3 real (2 High, 1 refute-as-deferred); reproduced, ranking-preserving  
**Location:** src/memorydb/vector.py:25-28 (_cosine), 53 (per-row call); fix at src/memorydb/store.py:110-116 (set_embedding)  

_cosine() recomputes `v_norm = sqrt(sum(x*x for x in v))` for EVERY stored vector on EVERY query, though stored vectors are immutable between queries. The query norm q_norm IS hoisted once (line 44) but the stored side is not. At the documented 1e5-vector ceiling this is the single largest latency in the system (~700-1100ms/query of pure Python, machine-dependent); pre-normalizing vectors at store time cuts the cosine loop ~35-50%, and top-k ranking is identical (verified). The default HashingEmbedder already L2-normalizes its output, so v_norm is ~1.0 for every row in the default config, making the per-query sqrt scan entirely redundant. This sits on the planner hot path (planner.py:85 -> index.search) and is reachable from ask/explain/context. One skeptic refuted High as a documented/deferred tradeoff (TD-004 review note line 46-47 and impl-ledger I7 both name this exact optimization and route the full fix to the sqlite-vec spec), and the irreducible dot-product remains either way — so this is a large constant-factor win at the regime ceiling, not a complexity fix. Sustained High by 2/3 because it bites at the stated supported scale today.

**Fix:** Normalize the vector once in Store.set_embedding (divide by L2 norm before pack()), persist a flag/model marker, and reduce _cosine to a bare dot product (dot/q_norm, or fold q_norm out if the query is also normalized). Note the cross-backend metric-consistency concern tracked in the sqlite-vec spec (C6) — keep normalization consistent with the future ANN path.

### MR-5 — High/performance — dirty_nodes() materializes the entire dirty corpus (with full bodies) into one list — ~338MB for 1e5 nodes during reembed_all
**Votes:** 3/3 real (2 High, 1 Medium); reproduced via tracemalloc, fix validated  
**Location:** src/memorydb/store.py:128-130 (SELECT * into list); src/memorydb/embedding_pipeline.py:79-89 (refresh batches the already-materialized list), 91-94 (reembed_all marks all dirty)  

dirty_nodes() does `SELECT * FROM nodes WHERE embed_dirty=1` and returns a list of every row including the `body` column (capped at 2000 chars by both extractors). refresh() calls it then slices the in-memory list into batch_size=128 chunks, so batching bounds only the embedder's working set, not the O(corpus) row materialization. reembed_all() marks ALL nodes dirty then calls refresh(), loading the entire corpus into RAM before the first batch embeds. Measured peak ~150MB with empty bodies, ~330-370MB with realistic 2000-char bodies for ~104k nodes — and the embedding DefaultSerializer (embedding_pipeline.py:27-59) never reads `body`, so the largest column is loaded purely to be discarded. Reachable via refresh_embeddings(full=True), the CLI --full flag, and a plain full index (which leaves the whole corpus dirty); the error messages at api.py:127/133 actively recommend full=True. Streaming via the existing idx_nodes_dirty partial index (SELECT id only, fetch each batch lazily) was verified to cut peak to ~5-15MB (24-73x). One skeptic rated Medium (330MB does not OOM typical CI/dev hardware at exactly 1e5, worst-case path is rare), but linear-in-corpus peak that defeats the explicit batch_size design and grows unbounded sustains High by 2/3.

**Fix:** Stream the dirty set: SELECT only id (using idx_nodes_dirty) and process batch-by-batch, fetching each batch's full rows lazily, keeping peak at O(batch_size). At minimum exclude `body` from the dirty_nodes projection since the embedding serializer never reads it.

### MR-6 — High/consistency — uid scheme NOT identical across adapters: PythonResolver omits CodeAdapter's #start_byte disambiguation, so duplicate-qualname symbols collide and precise edges land on the wrong def (@overload stub, not impl)
**Votes:** 2/3 real-High, 1 refute-to-Low (documented deferral); reproduced end-to-end  
**Location:** src/memorydb/adapters/code/python_resolver.py:122,153 (bare uid), 136,180-181 (module_defs.setdefault first-wins); adapters/code/__init__.py:133-138 (uid_for #start_byte); spec python-precise-resolver.md:56-57,147-149 (contradictory claims)  

Both adapters claim an IDENTICAL relpath::qualname uid scheme, but CodeAdapter.uid_for appends `#start_byte` to disambiguate repeated qualnames in a file while PythonResolver emits bare `f'{rel}::{qual}'` for every def, so same-qualname defs collide to ONE uid. Consequences (reproduced end-to-end on the typing.overload pattern: @overload stubs + impl all named `process`): (1) the two adapters produce different node-uid SETS for the file, so the precise-supersedes-coarse MAX merge cannot apply to the duplicated symbol; (2) the indexer dedups-by-uid keeping CodeAdapter's first node and dropping the rest, orphaning the later #start_byte nodes (e.g. the real impl); (3) module_defs.setdefault is first-wins so the precise CALLS edge targets the FIRST def = the line-3 @overload STUB, and locate('process') reports the call hitting the stub while the real impl is orphaned. The spec contradicts its own 'identical' claim at lines 147-149, deferring #start_byte. No test covers duplicate qualnames, which is why 79 tests pass. (This finding subsumes the original F22 duplicate-name/first-wins issue.) One skeptic refuted High as a documented-deferred tradeoff that points at a same-named same-file def (within-file precision gap, not a wrong unrelated symbol); 2/3 sustained High given silent wrong-node LOCATE attribution of a 1.0/0.97 edge plus orphaned real implementations on the common overload idiom.

**Fix:** Make the schemes truly identical: either give PythonResolver the same per-file collision disambiguation (deterministic suffix matching CodeAdapter's byte offset), or drop #start_byte from CodeAdapter and use a shared #<ordinal> disambiguation in BOTH so duplicates merge consistently. Also fix module_defs/class_methods first-wins to deprioritize @overload stubs (body is `...`) in favor of the real def, and reconcile the contradictory spec/docstring 'identical' claims (python_resolver.py:7, spec:56-57 vs :147-149).

### MR-7 — Medium/correctness — Package __init__.py never targeted: `from .X import f` / `from pkg import f` compute pkg/X.py / pkg.py instead of __init__.py, and collide into wrong 0.97 edges when a same-named sibling module exists
**Votes:** 3/3 real, Medium (downgraded from High); both collisions reproduced  
**Location:** src/memorydb/adapters/code/python_resolver.py:35-36 (_module_relpath), 93-99 (_from_relpath), vs 87-91 (_package_parts strips __init__)  

_from_relpath and _module_relpath join dotted parts + '.py' and never consider the <dir>/__init__.py form, while _collect_defs correctly keys a package's top-level defs as pkg/__init__.py::f. So every import targeting a PACKAGE is mis-keyed (`from . import sibling`, `from .. import topfunc`, `from .subpkg import f`, absolute `from pkg import f`). Normally this only under-resolves (skipped edge, safe). But when a real sibling module with the same basename and a same-named symbol exists, the computed wrong uid EXISTS and a wrong high-confidence 0.97 edge is materialized — verified against live Python semantics (a package directory shadows a same-named sibling .py module). Reproduced two collisions end-to-end through the store. Breaks the documented 'safe by construction' invariant. Severity settled at Medium by 3/3: the wrong-edge case requires an uncommon layout, while the far more common no-decoy case degrades only to a silent recall gap.

**Fix:** When computing an import target relpath, also accept the package form: probe both <parts>.py and <parts>/__init__.py (or have the indexer treat a directory target as its __init__.py). Minimum: resolve `from . import X` / `from .pkg import f` against <dir>/__init__.py.

### MR-8 — Medium/correctness — Nested function/class shadowing a module-level name produces a WRONG 1.0 edge: enclosing call resolves to the module def, not the local nested def
**Votes:** 3/3 real, Medium (downgraded from High); reproduced for CALLS and INHERITS  
**Location:** src/memorydb/adapters/code/python_resolver.py:208 (_scope_locals excludes is_namespace symbols), 178-181 (_resolve checks module_defs)  

_scope_locals records a name only if `is_parameter() or (is_local() and not is_namespace() and not is_imported())`. A name bound by a NESTED def/class is a namespace symbol, so it is NOT recorded as a local of the enclosing scope. _resolve then checks module_defs first, so a call to that name from the enclosing function resolves to the MODULE-level def at confidence 1.0, not the nested def that actually shadows it. The nested def node (outer.helper) exists, so the wrong uid is real and the edge materializes (the existence check passes). Verified via Python runtime that outer() actually invokes the nested helper. Generalizes to nested-class shadowing (wrong INHERITS @1.0). Settled Medium by 3/3: real maximum-confidence wrong edge, but bounded to same-file nested-def/class name collisions, no crash/corruption.

**Fix:** Record nested-def/class names as locals of their enclosing scope (do not exclude namespace symbols when deciding shadowing — the is_imported exclusion already separately handles the import case), or in _resolve prefer a same-file nested-qualname def (<stack>.<name> walked outward) over module_defs. Demote to skip if ambiguous.

### MR-9 — Medium/correctness — symtable scope-locals mis-keyed for same-named sibling scopes (last-wins dict overwrite) — shadowing protection silently fails, wrongly allowing or skipping
**Votes:** 3/3 real (1 High, 2 Medium); both failure directions reproduced incl. property/setter  
**Location:** src/memorydb/adapters/code/python_resolver.py:213 (out['.'.join(stack)] = locs), 177-179 (_resolve reads by colliding qualname key)  

_scope_locals keys its map by the dotted ast-style stack. When two sibling scopes share a name (legal Python: redefinition, conditional def, @overload, @property/@x.setter pairs, two lambdas), symtable yields two child tables both named e.g. `dup`/`outer.dup`, and the second overwrites the first under that key. _resolve reads the MERGED (wrong) local set, so depending on definition order it either wrongly ALLOWS a shadowed call (false 1.0 edge to a shadowed module def) or wrongly SKIPS a legitimate call (missing edge). Both directions reproduced, including an idiomatic @property getter + @setter case that materializes a wrong real.py::C.val -> real.py::helper @1.0. This defeats the very purpose of the symtable integration and violates the spec's 'no name-collision false positives within resolvable scope' Done criterion. Originally raised High; 2 of 3 skeptics downgraded to Medium (bounded to same-name sibling collisions; one wrong/missing edge per collision, no systemic corruption), so settled Medium.

**Fix:** Disambiguate scopes by (name, lineno) using table.get_lineno() on both the symtable side and the ast side, or walk symtable and ast in lockstep. At minimum, when a scope name is non-unique among siblings, conservatively union the locals of all same-named siblings (skip if shadowed in ANY).

### MR-10 — Medium/correctness — `import a.b.c` binds alias `a` to relpath a/b/c.py (not module a) → `a.run()` emits a wrong 0.95 edge to a/b/c.py::run
**Votes:** 3/3 real, Medium (downgraded from High); reproduced end-to-end  
**Location:** src/memorydb/adapters/code/python_resolver.py:104-106 (_collect_imports maps alias->full dotted path), 188-192 (_resolve attribute branch)  

For `import a.b.c`, _collect_imports sets alias = a.name.split('.')[0] = 'a' (correct binding name) but maps it to _module_relpath(a.name) = 'a/b/c.py' (wrong file — full dotted path). Python binds `a` to the top-level package (a.py or a/__init__.py); `a.b.c` only ensures the submodule is imported. So `a.run()` resolves to a/b/c.py::run instead of a/__init__.py::run, and when a/b/c.py defines run a wrong 0.95 edge materializes (verified against live Python; existence check passes because both endpoints exist). Bug is specific to unaliased multi-segment import + single-hop `a.attr()`: the aliased form `import a.b.c as abc; abc.run()` is correct, and the deep form `a.b.c.run()` is correctly skipped (recv is an Attribute, not a Name). Settled Medium by 3/3: concrete high-confidence false positive but narrow trigger conjunction, advisory retrieval-quality degradation not corruption.

**Fix:** For unaliased multi-segment `import a.b.c`, map the top-level binding name `a` to the top-level module path (_module_relpath(a.name.split('.')[0])), not the full dotted name. (Since _module_relpath yields x.py never x/__init__.py, a package top-level then safely yields a skipped edge rather than a wrong one — still the desired outcome.)

### MR-11 — Medium/correctness — `from . import submodule` then `submodule.func()` is never resolved (imported submodule modeled as a symbol, not a module)
**Votes:** 2/3 real-Medium, 1 refute-to-Low (documented best-effort); reproduced on this repo's own source  
**Location:** src/memorydb/adapters/code/python_resolver.py:107-115 (records 'sym' import), 188-194 (_resolve only handles 'mod' for attributes)  

`from . import util` (a very common pattern importing the SUBMODULE util) is recorded as ('sym', 'pkg.py', 'util') — wrong file and wrong kind. In _resolve, `util.helper()` has recv='util' which is a 'sym' not a 'mod', so the attribute call is skipped entirely. This under-resolves the entire `from . import mod; mod.func()` idiom (used 3+ times in this repo's own source, e.g. api.py:18 `from . import query as Q`; every Q.references_to() call drops its edge to the real query.py node). Safe by construction (no wrong edge) but a large, systematic coverage hole that compounds with MR-7 (the same line also mis-targets the bare name). One skeptic refuted as the documented best-effort/safe-by-construction tradeoff with tree-sitter as the fallback; 2/3 sustained Medium as it disables exactly the cross-module call resolution the precise resolver exists to provide.

**Fix:** When `from <pkg> import name` and no symbol disambiguation is available, also register `name` as a 'mod' candidate pointing at <pkgdir>/name.py (or its __init__.py), so `name.attr()` can resolve as a module attribute — possibly at slightly lower confidence to reflect the submodule-vs-reexport ambiguity.

### MR-12 — Medium/correctness — Mixed-dimension vector corpus searched without error via zip-truncation — silent garbage cosine scores reachable from the public facade
**Votes:** 3/3 real (2 Medium, 1 Low); reproduced via full facade incremental reindex  
**Location:** src/memorydb/vector.py:25-28 (_cosine zip(q,v)) + 53 (search, no dim filter); embeddings table stores `dim` (store.py:112)  

_cosine uses zip(q,v) which silently truncates to the shorter vector, producing a numerically valid but meaningless score with no error (and v_norm is computed over all of v, so it isn't even a true cosine on the overlap). I14 deferred this as 'covered by facade warn + I2 guard', but that coverage does not hold on the incremental path: the I2 guard only enforces dim consistency WITHIN one embed batch; refresh()/index() only re-embed embed_dirty nodes; and _check_embedder_compat only WARNS on a dim change, it does not block. So a user who switches to a different-dim embedder and runs an incremental index() (or refresh_embeddings without full=True) gets a mixed-dim corpus, and db.explain()/db.ask() run cosine across dim-64 and dim-32 vectors returning bogus seeds with no exception — reproduced end-to-end through the facade. Settled Medium by 3/3 (one argued Low: requires a warned-against deliberate misuse), but it fails open instead of loud on a public-API-reachable path.

**Fix:** Filter the SQL to `e.dim = len(query)` (embeddings already stores dim) so only matching-dim vectors are scanned, or raise in _cosine/search on a length mismatch, or gate search() so a corpus with >1 distinct dim raises a clear error. Re-classify I14 from 'covered' to 'open on the incremental path'.

### MR-13 — Medium/correctness — Over-deep / package-escaping relative imports not detected; truncated to a top-level path that can collide into a wrong 0.97 edge
**Votes:** 2/3 real-Medium, 1 refute-to-Low (invalid-Python trigger, spec cover); fix predicate corrected to node.level>len(pkg)  
**Location:** src/memorydb/adapters/code/python_resolver.py:93-99 (_from_relpath bounded slice with `else []` fallback)  

_from_relpath clamps an over-deep relative import (`from ..x` in a depth-1 package, `from .x` in a repo-root module, etc.) to a top-level path like x.py instead of signalling that it escapes the top package (an ImportError in real Python). If a real top-level x.py with symbol f exists, a wrong 0.97 edge is emitted; the most realistic trigger is an indexed-root vs package-root mismatch where a legitimate `from ...util import f` truncates onto an in-root util.py. Reproduced end-to-end through the indexer. Note the guard is off by one: the correct escape predicate (verified against CPython) is `node.level > len(self.pkg)` — the finding's suggested `node.level - 1 > len(self.pkg)` is insufficient. One skeptic refuted as triggering only on already-invalid/unimportable Python with spec cover ('resolve against repo_root; unresolved -> skip'); 2/3 sustained Medium.

**Fix:** When `node.level > len(self.pkg)` (escapes the top package) return None, so the bounded slice can no longer collide an invalid import into a real top-level module.

### MR-14 — Medium/performance — _resolve_pending is a per-row N+1: each resolved edge re-resolves dst_name and re-looks-up uid->id (5 statements/edge), no memoization
**Votes:** 2/3 real-Medium, 1 refute-to-Low (indexed lookups, embeddings dominate); reproduced 1.8-2.2x win  
**Location:** src/memorydb/indexer.py:259-267 (Python loop), 269-273 (_resolve_name 1 query/row), 275-280/store.py:82-108 (upsert_edge: 2 id_for + edge upsert + embed_dirty UPDATE)  

After the indexed candidate SELECT, the Python loop calls _resolve_name(dst_name) once per matched row and on a unique match calls upsert_edge = id_for(src)+id_for(dst)+edge upsert+embed_dirty UPDATE (~5 statements/edge). The candidate set is `src_file IN changed OR dst_name IN affected_names`; on a hub-name change affected_names is large and the matched pending set scales with references-to-that-name across the repo, with no per-name memoization and no uid->id cache. Measured ~18-28us/row (~356-556ms for 20k resolved edges); a drop-in memoization fix gave 1.8-2.2x with identical up/un counts. One skeptic refuted to Low (every lookup is an indexed point lookup, embeddings dominate per the spec, sub-second at the supported scale — optional micro-opt violating no contract); 2/3 sustained Medium as the dominant indexing cost on a hub-name change.

**Fix:** Memoize _resolve_name by dst_name within a run (many rows share a target name); cache uid->id (src and dst) in a per-run dict instead of two id_for queries per edge; batch the embed_dirty UPDATEs. Optionally resolve all unambiguous names in one set-based join rather than a Python loop.

### MR-15 — Medium/performance — Each changed file is opened and fully read 3-4 times per index run (size-check, binary-sniff, hash, then once per extractor)
**Votes:** 2/3 real-Medium, 1 Low (page-cache served); reproduced 3x amplification  
**Location:** src/memorydb/indexer.py:33-40 (skip_file sniff), 94 (full hash read, discarded), 172-194 (_extract_all); python_resolver.py:55, adapters/code/__init__.py:106 (each extractor re-reads); ports.py:28 (extract(path), not bytes)  

For every changed file the bytes are read repeatedly: skip_file does getsize + read(4096) sniff during discovery; the hash loop reads the FULL file to sha256 it then discards the bytes; _extract_all then calls ex.extract(path) for each handling extractor, and both PythonResolver and CodeAdapter independently re-open and read the full file. A Python file handled by both is fully read at least 3 times (hash + 2 extractors) plus the sniff. The Extractor port takes a path not bytes, structurally forcing the re-reads. Measured ~3x read amplification (e.g. 251KB read for an 82KB file). One skeptic rated Low (files capped at 1MB, re-reads hit the warmed OS page cache so the marginal cost is in-RAM copies + parse CPU, not disk seeks); 2/3 sustained Medium given the full 3-4x penalty on an initial/full index of a large repo.

**Fix:** Read the file bytes once in the hash loop (which already has them) and pass bytes to an `extract_bytes(path, data)` extractor entry point; fold the binary-sniff/size-check into that single read. Keeps I/O at 1x per changed file.

### MR-16 — Medium/ci — CI package smoke job cannot catch a dropped/renamed adapters subpackage — the one required for core Python indexing
**Votes:** 3/3 real (2 Medium, 1 Low); reproduced by deleting adapters from an installed wheel  
**Location:** .github/workflows/tests.yml:62-69 (wheel smoke: import memorydb, --help x2)  

The non-editable `package` job is the ONLY job that catches packaging/find-packages gaps (the editable test job keeps src/ on the path). Its smoke steps — `import memorydb`, `memorydb --help`, `memorydb-eval --help` — none transitively import memorydb.adapters (the adapter imports are lazy inside ExtractorRegistry.default(), reached only at open()/index() time). But memorydb.adapters.code.python_resolver is REQUIRED even for core-only Python indexing (api.py:35,46). Verified end-to-end: built a wheel, deleted site-packages/memorydb/adapters to simulate a find-packages regression — all three smoke steps still PASS, yet MemoryDB.open(':memory:') then fails with ModuleNotFoundError. So a future packaging regression would ship a green-CI wheel that breaks every real `pip install memorydb` + open/index. This is exactly the regression class the package job exists to guard, with a blind spot over the one subpackage core indexing depends on. One skeptic rated Low (the wheel ships adapters correctly today, so this is defense-in-depth not a present defect); 2/3 sustained Medium given total blast radius if it ever triggers.

**Fix:** Add to the wheel smoke step: `python -c "import memorydb.adapters.code.python_resolver, memorydb.adapters.code; from memorydb import MemoryDB; db=MemoryDB.open(':memory:'); db.index('<tmp py dir>'); db.close()"` — exercises the real lazy import + indexing path a true install hits.

### MR-17 — Low/consistency — context() / ask(as_context=True) packs the non-seed subgraph by node id, not uid — non-deterministic across re-index (R3L-4 fix missed the facade)
**Votes:** 3/3 real, Low; reproduced through public API  
**Location:** src/memorydb/api.py:198-204 (_explain_blocks: `seeds + sorted(nodes)`, keyed by node id); eval/__init__.py:171 sorts the same remainder by uid  

R3L-4 fixed churn-on-reindex non-determinism by ranking the non-seed EXPLAIN remainder by churn-invariant uid in vector.py and eval._explain_ranking, but the user-facing facade packer _explain_blocks still orders the remainder by node id. Node ids are not stable across re-index (nodes.id is INTEGER PRIMARY KEY with no AUTOINCREMENT, schema.sql:7; the indexer delete+reinserts reusing freed rowids), so a uid maps to different ids run-over-run. context() and ask(as_context=True) therefore produce different block orderings (and a history-dependent truncation boundary) for identical content, contradicting the ContextResult determinism intent — the same defect class R3L-4 closed, left open on the public path. Reproduced through the real public API. Severity: one skeptic argued this is the same Low class R3L-4 itself was triaged as, on an explicitly-placeholder serializer (cosmetic ordering of an LLM context string, no data loss/wrong results); seeds are always first and deterministic. Settled Low.

**Fix:** In _explain_blocks order the non-seed remainder by node uid (churn-invariant), mirroring eval._explain_ranking: `rest = sorted((nid for nid in nodes if nid not in seedset), key=lambda nid: nodes[nid]['uid'])`.

### MR-18 — Low/concurrency — migrate() is a check-then-act race; migration 3's ALTER TABLE ADD COLUMN is not idempotent — concurrent first-open can fail with 'duplicate column' or 'database is locked'
**Votes:** 3/3 real, Low; both failure modes reproduced concurrently  
**Location:** src/memorydb/migrations.py:96 (user_version read once before lock), 101-113 (BEGIN, no re-check), 49-52 (_m3 ALTER without IF NOT EXISTS)  

migrate() reads PRAGMA user_version once and never re-checks after acquiring the per-migration write lock. Two processes opening the same fresh file DB concurrently can both snapshot user_version=0; one migrates to v4, then the other re-runs migrations from its stale snapshot. Migrations 1/2/4 use IF NOT EXISTS (idempotent) but _m3 runs `ALTER TABLE nodes ADD COLUMN file_uid` with no IF NOT EXISTS, raising 'duplicate column name: file_uid' (or, more commonly, 'database is locked'). Reproduced both failures across concurrent opens. This violates the schema-migrations spec's explicit promise (line 100: 'the second opener sees the bumped version and skips') and its own risk-register mandate. Settled Low by 3/3: single-process and :memory: usage (the dominant case) are safe; only concurrent first-time opens of one file DB during the v0->v4 window trigger it; transient (retry succeeds once at v4); cheap, well-scoped fix.

**Fix:** Re-read user_version inside the loop after BEGIN IMMEDIATE and skip already-applied versions (rollback+continue if user_version >= m.version). Independently make _m3 idempotent by checking pragma_table_info before the ALTER. Consider also setting a busy_timeout.

### MR-19 — Low/correctness — nDCG with graded gains ignores binary-relevant expected_uids absent from gains, in both DCG and the ideal pool
**Votes:** 2/3 real-Low, 1 refute (intended graded-overrides-binary); reproduced  
**Location:** src/memorydb/eval/__init__.py:71-84 (ndcg_at_k: rel(uid)=gains.get(uid,0.0); ideal pool `gains or {uid:1.0 for uid in exp}`)  

When gains is non-empty, any uid in expected_uids that is NOT a key in gains is treated as fully irrelevant (gain 0) in BOTH the achieved DCG and the ideal DCG. A case that grades the top few items and lists the rest as binary-relevant (a natural authoring pattern) scores those binary retrievals zero and excludes them from IDCG, distorting nDCG: ranked=['a','b'], expected=['a','b','c'], gains={'a':3.0} yields 1.0, blind to relevant-retrieved 'b' and relevant-missed 'c'. Reproduced. Latent: no current fixture combines gains with a broader expected_uids, and recall/MRR are computed independently. One skeptic refuted as intended-by-design (graded judgments override binary per a standard IR convention asserted in test_eval.py and the eval spec); 2/3 sustained Low as a silent metric error the moment such a case is authored, with the docstring not warning of the override. Settled Low.

**Fix:** Define relevance as the union: rel(uid)=gains.get(uid, 1.0 if uid in exp else 0.0), and build the ideal pool from all relevant uids. Alternatively validate that providing gains requires gains to cover every expected uid (reject set(expected)-set(gains) non-empty) so the override is intentional. At minimum document that a non-empty gains fully overrides expected for nDCG.

### MR-20 — Low/correctness — traverse() emits non-existent seed ids as depth-0 'reached' nodes, breaking its return contract
**Votes:** 2/3 real-Low, 1 refute-not-actionable; reproduced  
**Location:** src/memorydb/query.py:59-66 (recursive CTE base term SELECT value,0 FROM json_each(:seeds), no existence check)  

The CTE base term seeds `reach` directly from json_each(:seeds) without checking the id exists in nodes, so traverse() returns {'id': <phantom>, 'depth': 0} for any seed id with no real node — a docstring/contract violation (a phantom is not a node). The live path is safe: the only caller (planner.explain) feeds seeds from index.search which always returns real node ids, and downstream get_nodes/subgraph_edges filter by `id IN (...)`, dropping phantoms. The hazard is purely for future/direct callers of the public traverse() with arbitrary ids. One skeptic refuted as zero-impact-today with harm purely hypothetical; 3/3 nonetheless agreed the contract violation is real and the fix is cheap. Settled Low.

**Fix:** Constrain the base term to existing nodes: `SELECT n.id, 0 FROM json_each(:seeds) j JOIN nodes n ON n.id = j.value` (id is the PK, negligible cost).

### MR-21 — Low/correctness — File handle leaked in indexer hash-diff read (`open(path,'rb').read()` without a context manager)
**Votes:** 3/3 real, Low; reproduced  
**Location:** src/memorydb/indexer.py:94  

`data = open(path, 'rb').read()` opens the file but never closes it explicitly, relying on CPython refcounting. It raises ResourceWarning under -W error::ResourceWarning and leaks descriptors on non-refcounting runtimes (PyPy) or during exception unwind; on a large repo that risks the open-file ulimit. It is the ONLY un-context-managed file read in the codebase — every other site (incl. indexer.py:37, python_resolver.py:55, adapters/code/__init__.py:106) uses `with`, so this is an inconsistency, not house style. Reproduced (ResourceWarning at indexer.py:94). Settled Low by 3/3: on default CPython the handle drops promptly, real impact is latent (PyPy / strict-warnings CI).

**Fix:** `with open(path, 'rb') as fh: data = fh.read()` (matching the rest of the codebase). Optionally enable -W error::ResourceWarning for the test run.

### MR-22 — Low/performance — BruteForceVectorIndex full-sorts the entire scored list for a top-k slice instead of a heap (k<<n)
**Votes:** 3/3 real, Low; fix verified output-identical  
**Location:** src/memorydb/vector.py:57-58 (scored.sort(...); scored[:k])  

search() builds `scored` over all n rows then full-sorts O(n log n) and slices [:k] for a typical k of 5-10; heapq.nlargest is O(n log k). Measured ~7-9x faster on the isolated sort step at 1e5 (~87ms saved/query), but negligible against the dominant cosine cost (the finding itself notes this), so it matters mainly as a free win and as n grows past 1e5. The proposed fix was verified byte-identical to the current output across edge cases, preserving the R3L-4 uid tiebreak and the I13 negative-k clamp. Note: the impl-review ledger (I7) inaccurately claims heapq is already applied — it is not. Settled Low by 3/3.

**Fix:** Replace `scored.sort(...); scored[:k]` with `heapq.nsmallest(max(0,k), scored, key=lambda t: (-t[0], t[2]))` (nsmallest on negated score preserves the -score,uid ordering and the negative-k clamp). Import heapq.
